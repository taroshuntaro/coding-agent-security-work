# 01 エグゼクティブサマリー

[← 目次へ戻る](README.md)

コーディングエージェントは、単にコードを提案するツールではない。権限を与えると、ファイルを読み書きし、シェルコマンドを実行し、ネットワークへ接続し、MCPやクラウドCLIを介して外部システムを操作できる。

業務利用では、次の原則を採用する。これらの絶対条件は[00 レッドライン](00-red-lines.md)に集約している。

1. **最小権限を既定値にする**
   - 読み取りだけで済む作業には書き込み権限を与えない。
   - ワークスペース外へのアクセスを許可しない。
   - ネットワークは原則無効、または接続先を許可リスト化する。

2. **承認と隔離を別の制御として設計する**
   - 承認は「実行前に人が確認する仕組み」。
   - サンドボックス・コンテナ・VMは「実行後に到達できる範囲を制限する仕組み」。
   - どちらか一方だけで安全とは判断しない。

3. **標準的な業務開発では、コンテナ内でエージェントと開発ツールを実行する**
   - ホストへ共有するのは原則として対象プロジェクトだけとする。
   - エージェントのコマンドはコンテナ内で実行する。
   - bind mountされたプロジェクトへの変更は、ホストの実ファイルへ直接反映される。

4. **本番シークレットと本番操作権限を与えない**
   - `.env` をGit管理しないだけでは不十分である。
   - エージェントから読み取れる場所に本番資格情報を置かない。
   - 必要な場合も、開発専用・最小権限・短寿命の認証情報を使用する。

5. **MCP、プラグイン、Hooks、クラウドCLIを権限境界として扱う**
   - MCPサーバーは「便利な拡張」ではなく、外部システムへの権限を持つ実行主体として審査する。
   - ユーザーが任意に追加できる状態を、業務利用の既定値にしない。

6. **エージェントの変更は、人間がレビューしてCI/CDで検証する**
   - 本番反映、`git push`、マイグレーション適用、クラウド変更などは、エージェントから切り離した承認付きパイプラインで行う。
   - エージェントに直接本番デプロイさせない。

7. **リポジトリ内の設定や指示を、信頼済みポリシーとみなさない**
   - `AGENTS.md`、`CLAUDE.md`、`.codex/`、`.claude/`、`.mcp.json`、Hooks、Dev Container定義、Dockerfile、ビルドスクリプトは、リポジトリを書き換えられる主体が変更できる。
   - 組織の強制ポリシーは、リポジトリ外のmanaged settings、`requirements.toml`、MDM、コンテナ・VM、ネットワーク制御で実装する。

8. **設定の存在ではなく、実効性をテストする**
   - `.env`の読み取り、ワークスペース外書き込み、任意通信、未承認MCP、`git push`などが実際に拒否されることを、導入時と更新時に確認する。
   - fail-openとなる条件を把握し、機密案件ではfail-closedの外側境界を設ける。

標準的な顧客案件では、**レベル2（標準業務開発）**を最低基準とし、顧客コード・個人情報・営業秘密を扱う場合は**レベル3（機密案件）**を推奨する（[05 業務利用レベル](05-levels.md)）。
