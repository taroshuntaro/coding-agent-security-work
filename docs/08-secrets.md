# 08 シークレットと `.env` の扱い

[← 目次へ戻る](README.md)

## 8.1 基本原則

- `.gitignore` はGitへの誤コミット防止であり、エージェントからの秘匿機能ではない。
- エージェントがプロジェクトを読めるなら、プロジェクト内の `.env` も読める可能性がある。
- 環境変数も、`env`、`printenv`、子プロセス、デバッグログから読める。
- 読み取り専用マウントは改変を防ぐが、内容の読み取りは防がない。
- シークレットをプロンプトやチャットへ貼り付けない。

> [!WARNING]
> Claude CodeのBashサンドボックスは、既定のread動作で `~/.aws/credentials` や `~/.ssh/` を読み取れる。`denyRead` への明示追加が必要である（[11.1](11-claude-code.md)）。「サンドボックス有効」だけでは資格情報は守られない。

## 8.2 推奨順位

1. モック・ローカルサービスを使い、シークレットを不要にする
2. Workload Identity、OIDC、IAM Roleなどの一時認証
3. Secret Managerから必要時に短寿命トークンを取得
4. 案件専用の開発シークレットを限定マウント
5. コンテナ環境変数として注入
6. プロジェクト内 `.env`
7. ソースコード・Dockerfile・イメージへの埋め込み

下位ほど避ける。

## 8.3 必須ルール

- 本番資格情報を開発コンテナへ渡さない（[00 R2](00-red-lines.md)）
- `~/.ssh`、`~/.aws`、`~/.kube` を丸ごとマウントしない
- 案件専用・環境専用の資格情報を使用
- 有効期限とローテーション手順を定義
- ログのAuthorization、Cookie、JWT、接続文字列をマスキング
- pre-commit、PR、CIでシークレットスキャン
- 漏えい時はGitから消すだけでなく、直ちに失効・ローテーション
- `.env.example` にはキー名のみを記載

## 8.4 `.env` が必要な場合の最低基準

```text
- 開発専用値のみ
- Git管理しない
- ファイル権限を限定
- エージェントのRead deny対象にする
- エージェントが必要な作業だけ、個別に値を注入する
- ネットワーク接続先を開発環境へ限定する
- 固定値ではなく短寿命値を優先する
```

## 8.5 値の渡し方とログ

- コマンドライン引数へ直接シークレットを渡さない。プロセス一覧、シェル履歴、監査ログへ残る可能性がある。
- `set -x`、詳細デバッグ、HTTPダンプ、例外オブジェクト、テストスナップショットに秘密値が出ないようにする。
- 可能であれば、短寿命トークン、標準入力、権限制限されたファイル、OSの資格情報ブローカーを使用する。
- Secret Managerから取得した値をワークスペース、ホームディレクトリ、パッケージ設定、キャッシュへ書き残さない。
- エージェントが実行したコマンドの標準出力・標準エラーは、モデルのコンテキストやセッション履歴へ入る可能性があるものとして扱う。

## 8.6 セッション履歴・メモリ・キャッシュ

シークレットや顧客情報は、ソースファイル以外にも残り得る。

- エージェントの会話・プロンプト履歴
- 自動メモリ
- コマンド出力・デバッグログ
- 差分・チェックポイント
- パッケージマネージャー・ビルドキャッシュ
- Codex Cloud / webの環境キャッシュ
- IDE・ターミナル履歴

案件ごとに保存先を分離し、暗号化、アクセス制御、保持期間、削除手順を定義する。保持が不要な案件では、製品が提供する履歴・メモリ・セッション永続化の無効化を検討する（Codex `history.persistence`、Claude Code `--no-session-persistence` 等。[付録C](appendix-c-volatile-values.md)）。
