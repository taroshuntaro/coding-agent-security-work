# コーディングエージェント業務利用 セキュリティ・運用ガイド

- **発行日**: 2026-06-20
- **最終更新日**: 2026-06-24
- **仕様確認基準日**: 2026-06-24
- **対象製品**: Codex（CLI・IDE連携・Codex web）、Claude Code（CLI・IDE連携）
- **対象読者**: コーディングエージェントを業務開発で利用する組織のセキュリティ管理者、開発リーダー、および利用者
- **目的**: コーディングエージェントに与える権限を、案件の機密性と影響度に応じて段階的に設計する
- **前提**: 特定のIDE・エディタには依存しない。VS Code、Eclipse、JetBrains系IDE、ターミナルエディタなどで同じ考え方を適用する

> [!IMPORTANT]
> 本ガイドのレベル区分は、OpenAIまたはAnthropicの公式区分ではなく、業務導入のために整理した組織向けモデルである。
> 製品の設定項目・既定値・提供条件は更新されるため、導入時には必ず最新の公式ドキュメントと契約条件を確認すること。変動しやすい具体値は[付録C 揮発値一覧](appendix-c-volatile-values.md)に集約している。

## 本ガイドの位置づけと強制力

本ガイドは、各案件チームが統制を設計するための**技術ガイドライン**である。本文の大半は「推奨（SHOULD）」であり、各案件の状況に応じて調整してよい。

ただし、**プラン・製品・案件を問わず必ず守るべき不変条件（MUST）**は別に定義する。これらは「推奨」ではなく、違反した場合は作業停止・是正の対象とする。

> **まず[00 絶対遵守（レッドライン）](00-red-lines.md)を読むこと。** ここに反する構成は、他のどの統制を満たしていても業務利用と認めない。

## スコープ：製品非依存層と製品固有層

本ガイドは構造的に2層からなる。

| 層 | 章 | 性質 |
|---|---|---|
| **製品非依存層** | 00〜09章、12〜20章、付録A・B | 原則・レベル・コンテナ・シークレット・統制目標。**Codex/Claude Code以外のコーディングエージェント（Cursor、GitHub Copilot、Gemini CLI 等）にもそのまま適用できる** |
| **製品固有層** | [10 Codex](10-codex.md)、[11 Claude Code](11-claude-code.md)、付録C | この2製品の設定キーに固有の実装例 |

Codex/Claude Code以外の製品を使う場合は、製品非依存層をそのまま適用し、[19 最終的な判断基準](19-final-criteria.md)の17問を各製品の機能へマッピングして用いる。他製品で本ガイドの統制目標を達成できない場合は、その不足を外部境界（コンテナ・VM・ネットワーク・OSポリシー）で補うか、利用可否を再評価する。

## 軸1：契約プラン（個人系 / チーム・ビジネス系）

統制の**達成手段**は契約プランに依存する。本ガイドは両系統を対象とし、どのプランの読者も利用できる。

- **個人系プラン**: 管理者による強制設定（Codexのmanaged requirements、Claude Codeのmanaged settings）や組織監査ログが**使えない**。設定は利用者自身のアカウント設定であり、技術的には**利用者本人がいつでも解除できる**。
- **チーム・ビジネス系プラン**: 管理者が**利用者本人に解除させない強制**をかけられ、組織監査ログを利用できる。

> [!IMPORTANT]
> **原則：統制目標（何を遮断すべきか）はレベルで決まり、プランに依存しない。達成手段がプランに依存する。**
>
> 個人系プランで上位レベル相当の遮断を実現するには、製品内強制の欠落を**製品の外側の境界（VM／コンテナ／OS・MDMポリシー／ネットワークプロキシ／短寿命認証／受入テスト）で完全に肩代わり**する。
>
> ただし**個人系プランは利用者本人が統制主体を兼ねる**。顧客契約や規制が「**第三者（管理者）による解除不能な強制**」を要求する案件では、外部境界を足しても要件を満たせないことがある。その場合はプランのアップグレード、または案件の辞退を検討する。

各プランで利用できる具体的な強制機能と代替手段は、[10 Codex](10-codex.md)・[11 Claude Code](11-claude-code.md)のプラン別節に示す。プラン名と機能対応は変動するため[付録C](appendix-c-volatile-values.md)を参照する。

---

## 軸2：業務利用レベル（L0〜L4）

案件を機密性と到達範囲で評価し、L0〜L4に分類する。**機密性軸と到達範囲軸の2軸で評価し、高い方を採用する。** 判定の詳細は[05 業務利用レベル](05-levels.md)、俯瞰は[06 レベル別早見表](06-quick-reference.md)を参照する。

標準的な顧客案件では**レベル2（標準業務開発）**を最低基準とし、顧客コード・個人情報・営業秘密を扱う場合は**レベル3（機密案件）**を推奨する。

## 本ガイドの読み方

1. [00 レッドライン](00-red-lines.md)で絶対条件を確認する。
2. [05 レベル](05-levels.md)で案件のレベルを2軸判定する。
3. 自組織の契約プラン（個人系／チーム・ビジネス系）を確認する。
4. [06 早見表](06-quick-reference.md)でレベル×プランの統制概要を把握する。
5. 使用製品の章（[10](10-codex.md)／[11](11-claude-code.md)）で具体的な設定とプラン別の達成手段を確認する。
6. [15 受入テスト](15-acceptance-tests.md)で設定が**実際に拒否される**ことを検証する。
7. [16 チェックリスト](16-pre-adoption-checklist.md)で導入可否を判断する。
