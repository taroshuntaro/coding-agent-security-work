# 04 IDE・エディタに依存しない実行構成

[← 目次へ戻る](README.md)

## 4.1 推奨構成A: エディタが開発コンテナへ接続する

```text
ホストOS
├─ エディタのUI
├─ コンテナランタイム
└─ 開発コンテナ
   ├─ Codex / Claude Code
   ├─ Git
   ├─ 言語ランタイム・SDK
   ├─ Lint・テスト・ビルドツール
   └─ プロジェクト
```

エディタがDev Container仕様やリモート開発に対応している場合に適する。エディタの画面はホストで動作するが、ターミナル、エージェント、ビルド、テストはコンテナ内で実行する。

## 4.2 推奨構成B: エディタはホスト、エージェントだけコンテナで実行する

```text
ホストOS
├─ 任意のエディタ
├─ project/  ───── bind mount ────┐
└─ コンテナランタイム             │
                                  ▼
                           開発コンテナ
                           ├─ /workspace/project
                           └─ Codex / Claude Code
```

IDEがコンテナ接続に対応していなくても採用できる。ホストのエディタで編集し、エージェントは `docker exec`、Compose、専用ランチャーなどからコンテナ内で起動する。

注意点:

- プロジェクトは複製されず、通常は同じホストファイルを参照する。
- コンテナ内の変更は、ホスト上のプロジェクトへ直接反映される。
- ホスト側IDEが実行するビルド、拡張機能、言語サーバーはコンテナ境界の外にある。
- エージェントの影響範囲は限定できるが、プロジェクト内のファイル破壊までは防げない。

## 4.3 推奨構成C: 専用VM・クラウド開発環境へリモート接続する

```text
開発者端末
└─ エディタ / SSH / Remote Development
   ↓
専用VM・VDI・クラウド開発環境
└─ コンテナまたは直接実行
   ├─ Codex / Claude Code
   └─ プロジェクト
```

顧客機密や高機密案件に適する。ソースコードやシークレットを開発者端末へ残さず、接続先を集中管理できる。

## 4.4 リポジトリが定義する開発環境を自動的に信頼しない

次のファイルは、コンテナを利用するための設定である一方、ビルド時・起動時・接続時にコードを実行できる（[00 R6](00-red-lines.md)）。

- `.devcontainer/devcontainer.json`
- `Dockerfile`、Composeファイル
- `postCreateCommand`、`postStartCommand`などのライフサイクルコマンド
- Maven/Gradle/npmなどのプラグイン・lifecycle script
- Git Hooks、Claude Code Hooks、Codex Hooks
- `.mcp.json`、エージェント設定、プロジェクト指示ファイル

初回チェックアウト直後に自動ビルド・自動接続せず、内容をレビューする。特に、Dockerソケット、ホストディレクトリ、SSH Agent、クラウド資格情報、デバイスをマウントする定義を要審査とする。

コンテナ内でエージェントを動かしても、**コンテナを作成する処理自体**がホストのDocker Engineやビルド機構で動く点に注意する。

## 4.5 Windows環境の考え方

製品組み込みサンドボックスの対応範囲はOSごとに異なる。特にClaude CodeのBash sandboxはmacOS、Linux、WSL2が対象であり、Windowsネイティブ環境とWSL1を同等に扱わない。

Windows中心の開発では、次のいずれかを標準化すると管理しやすい。

- WSL2内でエージェントを実行する
- Docker Desktop等のLinuxコンテナ内で実行する
- 専用Linux VM・クラウド開発環境へ接続する

Codexについても、ネイティブWindows、WSL、コンテナで実装・制約が異なるため、対象OSごとにポリシーテストを行う。
