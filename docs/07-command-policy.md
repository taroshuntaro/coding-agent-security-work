# 07 共通の推奨コマンドポリシー

[← 目次へ戻る](README.md)

プロジェクトごとに実際のコマンドを棚卸しし、次のように分類する。

## 7.1 自動許可の候補

- `git status`
- `git diff`
- `git log`
- プロジェクト内の検索
- Lint
- 単体テスト
- 型チェック
- ローカルビルド
- フォーマット
- 読み取り専用のコード生成確認

自動許可する場合も、コマンドが任意スクリプトを間接実行しないか確認する。たとえば `npm test` の実体はリポジトリ内の任意スクリプトである。

## 7.2 都度確認の候補

- 新しい依存パッケージの追加・更新
- `npm install`、`pip install`、Maven/Gradleの依存変更
- DBマイグレーション生成
- Dockerイメージのビルド
- `git commit`
- PR・Issueの作成
- 外部URLへのアクセス
- コード生成ツールの初回実行

## 7.3 原則拒否の候補

- `git push --force`
- `git reset --hard` を共有ブランチへ適用する操作
- `curl`、`wget` による任意送信
- `ssh`、`scp`
- `kubectl`、`helm` の変更操作
- `terraform apply`、`destroy`
- `aws`、`az`、`gcloud` の変更操作
- 本番DBクライアント
- デプロイコマンド
- シークレットの表示
- Dockerソケット経由のホスト操作
- 権限昇格、`sudo`
- ワークスペース外への削除・書き込み

拒否対象はコマンド名だけでなく、MCP、Hooks、スクリプト、CI/CD経由の同等操作も含める。

> [!WARNING]
> コマンド文字列のallow/denyパターンだけを、強いセキュリティ境界として扱わない。ラッパースクリプト、別名コマンド、シェル展開、インタプリタ経由、MCP、ビルドツール経由で同じ副作用を起こせる。
> 強い制御が必要な操作は、ファイルシステム、ネットワーク、資格情報、外部サービス側RBAC、コンテナ・VMで到達不能にする。
