# 09 コンテナの推奨設定

[← 目次へ戻る](README.md)

## 9.1 推奨

- 非rootユーザー
- 対象リポジトリだけをマウント
- 必要なマウントはread-onlyを優先
- root filesystemをread-onlyにし、必要な書込先だけVolumeまたはtmpfs化
- Linux capabilitiesを原則削除
- `no-new-privileges`
- CPU、メモリ、PID上限
- エグレス制御
- ベースイメージとツールバージョンの固定
- 定期的なイメージ更新と脆弱性スキャン
- エージェントの設定・認証用Volumeを案件単位で分離
- 不要なコンテナは破棄

## 9.2 禁止または要審査

```text
--privileged
--network host
-v /:/host
-v /var/run/docker.sock:/var/run/docker.sock
ホストのホームディレクトリ全体のマウント
ホストのSSH・クラウド・Kubernetes資格情報の一括マウント
USB・GPU・デバイスの無制限共有
```

Dockerソケット共有は、コンテナからホスト上の他コンテナやマウントを操作できる場合があり、実質的なホスト権限として扱う。

## 9.3 bind mountの性質

ホストのプロジェクトをbind mountした場合、コピーや後同期ではない。コンテナ内プロセスがマウント領域を変更すると、ホストの実ファイルが直接変更される。

```text
コンテナ内でスクリプト実行
  ├─ /usr、/tmp、コンテナ内設定の変更 → 原則コンテナ内
  ├─ /workspace/project の変更       → ホストのプロジェクトへ直接反映
  └─ ネットワーク先の変更             → 外部システムへ影響
```

したがって、コンテナはプロジェクト自体を保護するものではなく、主にプロジェクト外のホスト環境への影響を限定する。

## 9.4 コンテナ構成の受入条件

- リポジトリ内のDockerfile・Dev Container定義をレビュー済み
- Dockerソケット・ホストルート・ホーム全体を共有していない
- 起動コマンド、entrypoint、ライフサイクルスクリプトを確認済み
- イメージの取得元、digestまたはバージョンを固定
- コンテナからホストエディタの拡張機能や資格情報へ暗黙に到達できない
- コンテナ破棄後も必要なソース差分だけを安全に回収できる
