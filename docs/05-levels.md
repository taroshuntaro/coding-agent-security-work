# 05 業務利用レベル

[← 目次へ戻る](README.md)

## 5.0 レベルの決め方（2軸判定）

レベルは、**機密性**と**到達範囲**の2軸で評価し、**高い方を最終レベルとして採用する**（`最終レベル = max(機密性レベル, 到達範囲レベル)`）。片方の軸だけが高い案件（例：機密性は中程度だが本番DBに到達しうる障害調査）を、低く誤判定しないための規則である。

### 機密性軸

| 値 | 内容 |
|---|---|
| L1 | 公開情報・公開リポジトリ・機密性のない社内サンプル |
| L2 | 通常の社内システム、合意済み顧客開発（本番データを含まない） |
| L3 | 非公開の顧客ソースコード、営業秘密、認証・認可・決済・個人情報に関わるコード |
| L4 | 規制対象の高度な個人情報、金融・医療・公共・安全保障など追加規制データ |

### 到達範囲軸

| 値 | 内容 |
|---|---|
| L1 | 読み取り中心。書き込みも使い捨て環境に限定 |
| L2 | 開発・テスト環境への書き込み。本番には到達しない |
| L3 | 本番相当構成を扱うが本番データには到達しない |
| L4 | 本番データ・本番システムに近接、または高権限インフラを操作しうる |

### 判定例

- 公開リポジトリの調査 → 機密性L1・到達L1 → **L1**
- 合意済み顧客案件の通常実装（開発環境のみ） → 機密性L2・到達L2 → **L2**
- 顧客の認証コード改修（開発環境のみ） → 機密性L3・到達L2 → **L3**
- 機密性は中程度だが本番DBに触れうる障害調査 → 機密性L2・到達L4 → **L4**

> 契約プラン（個人系／チーム・ビジネス系）は**レベルを変えない**。レベルは案件の性質で決まる。プランは、そのレベルの統制を**どの手段で達成するか**を決める（[README「軸1：契約プラン」](README.md)）。

### 契約・管理基盤の前提

| レベル | 前提となる契約・管理基盤 |
|---|---|
| L1〜L2 | ローカル設定＋使い捨て／非rootコンテナで概ね成立。個人系プランでも実施可能 |
| L3 | **管理者による解除不能な強制が要件となる案件では、チーム・ビジネス系プラン＋管理配布基盤（MDM／managed設定）が前提**。個人系プランの場合は外部境界で代替し、第三者強制が必要な案件は実施しない |
| L4 | 上記に加え、専用VM・集中監査・DLPなどの組織基盤が前提。基盤が無ければ実施しない |

具体的なプラン名・機能対応は[付録C](appendix-c-volatile-values.md)を参照する。

## 5.1 レベル0: 非管理利用・業務利用不可

### 状態

- 個人アカウントを利用
- ホストOS上でfull accessまたは権限バイパス
- ホームディレクトリ全体へアクセス可能
- 任意のインターネット通信を許可
- 本番資格情報を保持
- 任意のMCP・プラグインを追加可能
- 監査ログなし
- エージェントから直接デプロイ可能

### 判断

業務利用には適さない。検証用の公開コードや使い捨て環境を除き、禁止状態として扱う。[00 レッドライン](00-red-lines.md)のR1〜R6に複数抵触する。

## 5.2 レベル1: 低リスク調査・試行

### 対象

- 公開情報・公開リポジトリ
- 機密性のない社内サンプル
- コードレビュー、調査、設計相談
- PoC初期段階

### 推奨制御

| 項目 | 推奨 |
|---|---|
| 実行環境 | ホスト上の組み込みサンドボックス、または使い捨てコンテナ |
| ファイル | 原則read-only。必要な場合のみ対象ワークスペースへ書き込み |
| 承認 | 標準・on-request。危険コマンドは都度確認 |
| ネットワーク | 組み込み検索は必要時のみ。シェル通信は原則無効 |
| シークレット | 与えない |
| MCP | 原則無効。必要なら読み取り専用・限定サーバー |
| Git | push・merge・releaseは禁止 |
| 人的確認 | 生成結果と差分を開発者が確認 |

### 不適切な用途

- 顧客コード
- 個人情報
- 本番障害調査
- 本番認証情報を必要とする作業

## 5.3 レベル2: 標準業務開発 — 組織の推奨既定値

### 対象

- 通常の社内システム
- 顧客との合意済み開発案件
- 本番データを含まない開発・テスト
- 一般的な実装、テスト、リファクタリング

### 推奨制御

| 項目 | 推奨 |
|---|---|
| 実行環境 | 非rootの開発コンテナ、または同等の隔離と権限制御を備えた管理環境 |
| ホスト共有 | 対象プロジェクトだけをbind mount |
| ファイル | ワークスペース書き込み可。ワークスペース外はdeny |
| 保護対象 | `.env`、資格情報、IDE設定、エージェント設定をdenyまたはread-only |
| 承認 | 標準・on-request。bypass/full accessは禁止 |
| ネットワーク | 原則無効。必要なレジストリ・Git・公式ドキュメントだけ許可 |
| 組み込みWeb機能 | 製品ごとに制御する。Codexは`cached`または`disabled`を優先し、Claude Codeは`WebSearch`と`WebFetch`を個別に制御する |
| シークレット | 開発専用・最小権限・短寿命のみ |
| MCP | 組織許可リスト。書き込み権限は原則与えない |
| Git | branch内の編集まで。push・PR作成は承認対象 |
| デプロイ | CI/CD経由。エージェントに直接権限を与えない |
| 監査 | 組織アカウント、PR、CIログ、利用状況を記録 |
| レビュー | 人間によるdiffレビューとテストを必須化 |

### コンテナの最低条件

- 非rootユーザー
- `--privileged` を使用しない
- ホストネットワークを使用しない
- Dockerソケットをマウントしない
- `~/.ssh`、`~/.aws`、`~/.kube`、ホームディレクトリ全体をマウントしない
- CPU、メモリ、プロセス数を必要に応じて制限
- ベースイメージと主要ツールのバージョンを固定
- コンテナイメージを脆弱性スキャン

詳細は[09 コンテナの推奨設定](09-containers.md)を参照。

## 5.4 レベル3: 顧客機密・営業秘密・重要システム

### 対象

- 非公開の顧客ソースコード
- 営業秘密
- セキュリティ設計
- 認証・認可・決済・個人情報に関連するコード
- 本番相当構成を扱うが、本番データは扱わない作業

### 推奨制御

レベル2に加えて、次を実施する。

| 項目 | 推奨 |
|---|---|
| 実行環境 | 組織管理の専用VM、VDI、クラウド開発環境、または強化コンテナ |
| 永続化 | 案件専用Volumeまたは管理された作業コピー。案件間で状態を共有しない |
| ネットワーク | 組織プロキシ経由。接続先・HTTPメソッドを許可リスト化 |
| 組み込みWeb機能 | 原則無効または許可先限定。Codexの`live`検索、Claude Codeの`WebSearch`・`WebFetch`は承認された調査に限定 |
| シークレット | Secret Manager・Workload Identity・短寿命トークン |
| 認証情報 | 本番用は禁止。開発・検証環境も案件専用かつ最小権限 |
| 設定 | 管理者が強制し、ユーザー・リポジトリから解除できないようにする |
| MCP・Hooks | 管理者定義のみ。任意追加を禁止 |
| 依存取得 | 社内ミラーまたは許可されたレジストリ。lockfileとバージョン固定 |
| 外部操作 | Git push、Issue作成、クラウド変更を承認対象または禁止 |
| 監査 | エージェント利用、設定変更、外部ツール呼び出しを監視 |
| データ | 顧客契約、DPA、保存期間、学習利用、リージョンを事前確認 |

> [!IMPORTANT]
> 「設定を管理者が強制し、ユーザー・リポジトリから解除できないようにする」は、**チーム・ビジネス系プラン＋管理配布基盤がある場合に成立する**。個人系プランではこの強制を製品内で作れないため、外部境界（専用VM・OS/MDMポリシー・ネットワーク・短寿命認証）で代替する。代替しても「第三者による解除不能な強制」が契約上要求される案件は、個人系プランでは実施しない（[00 R3・R6](00-red-lines.md)）。

### 追加推奨

- リポジトリを信頼済みと判断する前に、スクリプト、Hooks、MCP設定、エージェント指示ファイルを人間が確認する。
- `package.json` のlifecycle script、Makefile、Gradle/Mavenプラグイン、CI定義などを非信頼コードとして扱う。
- エージェントが生成したマイグレーションや認証変更には、セキュリティレビュー担当者を設定する。
- 顧客ごとにエージェント設定、認証情報、ログ、Volumeを分離する。

## 5.5 レベル4: 高機密・規制対象・本番隣接

### 対象

- 本番データや高度な個人情報に近接する環境
- 金融、医療、公共、安全保障など追加規制がある案件
- 高権限インフラコード
- 重大事故につながるシステム
- ソースコード持ち出しが厳格に制限される案件

### 原則

レベル4では、製品を使えることを前提にしない。リスク評価、契約、顧客承認、法務・セキュリティ審査を経て利用可否を決める。**必要な管理基盤を持たないプラン・組織では実施しない。**

### 推奨制御

| 項目 | 推奨 |
|---|---|
| 実行環境 | 使い捨て専用VM・マイクロVM・VDI。ローカル端末へコードを残さない |
| 分離 | 案件・タスク単位の環境。終了後に破棄 |
| ネットワーク | deny by default。推論API、認証、内部ミラーなど必要最小限 |
| Web・外部情報 | 原則無効。必要時は別の非機密端末で調査 |
| シークレット | 動的・短寿命。タスク単位。人間にも固定値を配布しない |
| 外部システム | 読み取り専用。書き込みはエージェントから分離 |
| 本番 | エージェントから到達不能。CI/CDにも独立した承認を要求 |
| データ | DLP、持ち出し制御、クリップボード・ダウンロード制御 |
| 監査 | 集中ログ、セッション記録、アラート、定期レビュー |
| 更新 | 承認済みバージョンのみ。自動更新を止め、段階的に検証 |
| 例外 | 期限・責任者・対象・理由を記録した例外申請制 |

### 実施できない場合

必要な隔離・契約・監査・データ処理要件を満たせない場合は、その案件ではコーディングエージェントを使用しない。
