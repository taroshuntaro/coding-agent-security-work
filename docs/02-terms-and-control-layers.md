# 02 用語と制御レイヤー

[← 目次へ戻る](README.md)

## 2.1 権限・承認

エージェントが、特定のツールやコマンドを実行してよいかを決定する。

例:

- ファイル読み取りは自動許可
- ファイル編集は許可
- `git push` は都度確認
- `kubectl`、`terraform apply`、`aws` は拒否

承認プロンプトは、人間によるレビュー機会を提供する。ただし、プロンプト疲れ、誤承認、包括的な恒久許可によって実効性が低下する可能性がある。

## 2.2 エージェント組み込みサンドボックス

エージェントが起動したコマンドについて、OSレベルなどでファイルシステムやネットワークへのアクセス範囲を制限する。

サンドボックスという名称であっても、必ずしも別OSや仮想マシン内で実行されるとは限らない。ホストOS上のプロセスに制約を掛ける方式もサンドボックスに含まれる。

## 2.3 コンテナ

エージェント、シェル、SDK、ビルドツール、言語サーバーなどを、ホストとは異なるプロセス・ファイルシステム・ネットワーク空間で実行する。

ただし、コンテナは通常ホストのカーネルを共有するため、VMと同等の隔離ではない。また、ホストのプロジェクトをbind mountすると、その範囲のファイルはコンテナから直接変更できる。

## 2.4 仮想マシン・マイクロVM・VDI

ゲストOSまたは専用の仮想化境界を設ける。コンテナより強い隔離が必要な案件で使用する。

## 2.5 ネットワーク制御

次の通信を分けて設計する。

- モデル推論・認証に必要な通信
- 組み込みWeb検索
- サンドボックス内コマンドのHTTP通信
- パッケージレジストリへの通信
- Gitホスティングへの通信
- MCP・Hooks・プラグインの通信
- 開発環境・本番環境への通信

「Web検索を許可すること」と「シェルから任意の `curl` を許可すること」は同じではない。

## 2.6 外部システム側の権限

GitHub、GitLab、AWS、Azure、Google Cloud、Kubernetes、DB、チケット管理、社内SaaSなどの権限は、ローカルサンドボックスとは別に評価する。

ローカルファイルへのアクセスが限定されていても、強いクラウド認証情報やMCP権限があれば、外部環境へ重大な変更を行える。

## 2.7 指示・拡張レイヤー

権限・サンドボックス・ネットワークが「実行を境界で止める」層であるのに対し、エージェントの挙動は**指示・拡張ファイル**によっても方向づけられる。これらは隔離境界ではないが、新たなコード実行経路・権限経路・安全指示の上書き経路を持ち得るため、コードと同様に統制する。

- **プロジェクト指示ファイル**（`CLAUDE.md`・`AGENTS.md` など）: セッション開始時または関連ディレクトリの作業時にロードされ、挙動を方向づける。リポジトリ由来のため信頼前にレビューし、所有者を明示してコードレビュー対象とする（[00 R6](00-red-lines.md)）。
- **パススコープ・ルール**（例: Claude Codeの `.claude/rules/`、`paths:` で適用範囲を限定）: ディレクトリ単位の指示。被書換のため同様にレビュー対象。allow/ask/denyの**権限ルールとは別物**で、強制ポリシーではない。
- **スキル・カスタムコマンド**: 名称・説明のみ常時ロードし本体は呼び出し時に動的ロードされる。インラインのシェル実行経路を持ち得るため、供給元審査・許可リスト・シェル実行の抑止を検討する（[13](13-mcp-plugins-hooks.md)）。
- **サブエージェント**: 独立コンテキストで起動し、**自らツールを呼び出す**。親セッションの権限・サンドボックス境界の継承を確認し、無人実行では権限増幅経路として棚卸しする。定義ファイルをレビュー対象とする。
- **出力スタイル・システムプロンプト追記**: 既定のシステムプロンプト（安全方針を含む）を**置換・追記**し、コンパクション後も保持され得る。安全指示の上書きに注意し、機密案件では既定指示を全置換する設定を避ける。

これらは**指示レイヤーであって隔離境界ではない**。プロンプト指示で確実に止めたい操作は、権限deny・サンドボックス・Hook（[13](13-mcp-plugins-hooks.md)）など決定論的な層で担保する。製品別の具体的な統制手段・設定キーの有無は[11.9](11-claude-code.md)・[付録C](appendix-c-volatile-values.md)に集約する。

## 2.8 制御プレーンとfail-open / fail-closed

管理ポリシーをクラウドから配信する場合、ネットワーク障害、キャッシュ破損、古いクライアント、設定取得失敗時の挙動を確認する。

- **fail-closed**: ポリシーを取得・適用できなければ起動または操作を拒否する
- **fail-open**: ポリシーを取得・適用できなくても既定値またはローカル設定で継続する

機密案件では、クラウド配信ポリシーだけに依存せず、OS管理ファイル、MDM、コンテナ・VM、プロキシ、認証情報の非配布など、取得失敗時にも残る統制を組み合わせる。

> [!NOTE]
> Codexのクラウド配信型managed requirementsは**fail-open**仕様である（[10.3.2](10-codex.md)）。Claude Codeは `sandbox.failIfUnavailable` や `forceRemoteSettingsRefresh` でfail-closed寄りに設定できる（[11.5](11-claude-code.md)）。いずれも[15 受入テスト](15-acceptance-tests.md)で実挙動を確認する。
