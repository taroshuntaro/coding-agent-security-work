# 02 用語と制御レイヤー

[← 目次へ戻る](README.md)

## 2.1 権限・承認

エージェントが、特定のツールやコマンドを実行してよいかを決定する。

例:

- ファイル読み取りは自動許可
- ファイル編集は許可
- `git push` は都度確認
- `kubectl`、`terraform apply`、`aws` は拒否

承認プロンプトは、人間によるレビュー機会を提供する。ただし、プロンプト疲れ、誤承認、包括的な恒久許可によって実効性が低下する可能性がある。

## 2.2 エージェント組み込みサンドボックス

エージェントが起動したコマンドについて、OSレベルなどでファイルシステムやネットワークへのアクセス範囲を制限する。

サンドボックスという名称であっても、必ずしも別OSや仮想マシン内で実行されるとは限らない。ホストOS上のプロセスに制約を掛ける方式もサンドボックスに含まれる。

## 2.3 コンテナ

エージェント、シェル、SDK、ビルドツール、言語サーバーなどを、ホストとは異なるプロセス・ファイルシステム・ネットワーク空間で実行する。

ただし、コンテナは通常ホストのカーネルを共有するため、VMと同等の隔離ではない。また、ホストのプロジェクトをbind mountすると、その範囲のファイルはコンテナから直接変更できる。

## 2.4 仮想マシン・マイクロVM・VDI

ゲストOSまたは専用の仮想化境界を設ける。コンテナより強い隔離が必要な案件で使用する。

## 2.5 ネットワーク制御

次の通信を分けて設計する。

- モデル推論・認証に必要な通信
- 組み込みWeb検索
- サンドボックス内コマンドのHTTP通信
- パッケージレジストリへの通信
- Gitホスティングへの通信
- MCP・Hooks・プラグインの通信
- 開発環境・本番環境への通信

「Web検索を許可すること」と「シェルから任意の `curl` を許可すること」は同じではない。

## 2.6 外部システム側の権限

GitHub、GitLab、AWS、Azure、Google Cloud、Kubernetes、DB、チケット管理、社内SaaSなどの権限は、ローカルサンドボックスとは別に評価する。

ローカルファイルへのアクセスが限定されていても、強いクラウド認証情報やMCP権限があれば、外部環境へ重大な変更を行える。

## 2.7 制御プレーンとfail-open / fail-closed

管理ポリシーをクラウドから配信する場合、ネットワーク障害、キャッシュ破損、古いクライアント、設定取得失敗時の挙動を確認する。

- **fail-closed**: ポリシーを取得・適用できなければ起動または操作を拒否する
- **fail-open**: ポリシーを取得・適用できなくても既定値またはローカル設定で継続する

機密案件では、クラウド配信ポリシーだけに依存せず、OS管理ファイル、MDM、コンテナ・VM、プロキシ、認証情報の非配布など、取得失敗時にも残る統制を組み合わせる。

> [!NOTE]
> Codexのクラウド配信型managed requirementsは**fail-open**仕様である（[10.3.2](10-codex.md)）。Claude Codeは `sandbox.failIfUnavailable` や `forceRemoteSettingsRefresh` でfail-closed寄りに設定できる（[11.5](11-claude-code.md)）。いずれも[15 受入テスト](15-acceptance-tests.md)で実挙動を確認する。
